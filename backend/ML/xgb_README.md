# `train.py` — 학습 파이프라인 안내

`train.py`는 **`train.csv`**만 사용해서 URL이 **정상(0)** 인지 **악성(1)** 인지 구분하는 **XGBoost 모델 하나**를 학습합니다. 아래는 **코드가 실제로 하는 일**을 **단계별로** 쉽게 풀어 쓴 설명입니다.

---

## 준비물

| 항목 | 설명 |
|------|------|
| **입력 파일** | 같은 폴더에 `train.csv` (컬럼: `ID`, `URL`, `label`) |
| **실행** | `train.py` |
| **생성 파일** | `xgb_final_v1.json` (모델), `xgb_train_artifacts.joblib` (전처리·맵 저장) |

**패키지 예시**

```bash
pip install pandas numpy scikit-learn xgboost tldextract category_encoders joblib
```

---

## 한 줄 요약: 학습이 가는 길

1. **라벨만** 읽어서 → 데이터를 **8 : 1 : 1**로 나눌 **행 번호**를 정한다.  
2. **Train 구간만** 다시 읽어서 → 그룹별 악성 비율(타깃 인코딩)을 **리키지 없이**(Val/Test 구간 데이터는 포함시키지 않음) 준비한다.
3. CSV를 **또 한 번** 읽으면서 → 각 행마다 **숫자 피처 + 문자 피처**를 채운다.  
4. 문자는 **이진 인코딩**, 숫자는 **표준화**한다. (Train에만 맞춤)  
5. **XGBoost**로 학습하고, **검증·테스트**로 성능을 본 뒤 파일로 저장한다.

## 단계별 설명 (`main()` 안 순서)

### 1단계 — “누가 Train, 누가 검증, 누가 시험인지” 먼저 정하기

**하는 일**

- `train.csv`에서 **`label` 컬럼만** 조금씩(`chunksize`) 읽어서, 전체 라벨 배열을 만듭니다.
- 그다음 `train_test_split`을 **두 번** 써서 행 번호를 세 덩이로 나눕니다.
  - **Train 80%** / 나머지 20%
  - 그 20%를 다시 반으로 → **Validation 10%**, **Test 10%**
- `stratify`로 **정상/악성 비율**이 세 덩이에 비슷하게 가도록 맞춥니다.

**왜 이렇게 하나요?**

- 파일이 매우 크면 **URL까지 한 번에** 읽기 어렵습니다. 그래서 **가장 가벼운 라벨만** 먼저 읽고, “몇 번째 행이 어느 집합인지”만 정해 둡니다.
- 나중에 URL을 읽을 때도 **같은 규칙**으로 Train/Val/Test에 넣을 수 있습니다.

**참고**

- `USE_SAMPLE = True`이면 전체 중 일부만 뽑아서 같은 비율로 나누는 빠른 테스트용입니다.

---

### 2단계 — “그룹별 악성 비율”을 Train 안에서만 안전하게 만들기

**하는 일**

- CSV를 다시 조금씩 읽으면서, **1단계에서 Train으로 정해진 행만** 골라서  
  `url_tld`, `url_sld`, `label`만 모아 표를 만듭니다. (`URL`에서 경로/도메인을 잘라낸 값입니다.)
- **Train 표**에 대해:
  - **K-Fold(기본 5번 접기)** 로 데이터를 나눕니다.(Ex. A, B, C, D, E 그룹)
  - 각 접기에서 “검증 쪽 행”에는 **“학습 쪽에만 있는 데이터”로 계산한** 그룹별 평균 라벨을 붙입니다.(Ex. A그룹 소속이라면, A를 뺀 [B, C, D, E]의 평균값을 가져와서 A URL 옆에 적어 넣습니다.)
  - 이걸 **OOF(Out-of-Fold)** 라고 부릅니다.
- **Validation·Test에 나중에 쓸 맵**도 여기서 만듭니다: **Train 전체**를 기준으로 `url_tld` / `url_sld`별 평균 라벨을 구해 둡니다.

**왜 이렇게 하나요?**

- “이 TLD면 악성 비율이 얼마다” 같은 숫자는 **라벨을 써서** 만듭니다.  
  그걸 **그 행의 라벨까지 포함해서** 한 번에 계산하면, 모델이 **정답을 미리 본 것처럼** 되어 성능이 부풀어질 수 있습니다(리키지).
- 그래서 **Train 안에서도** “이 행은 빼고 통계를 낸 값”을 붙이고(OOF), **Val/Test**에는 **Train에서만 본 통계**로만 채웁니다.

---

### 3단계 — “한 줄 한 줄” 피처를 채워 넣기

**하는 일**

- CSV를 **또 한 번** 위에서부터 읽습니다.
- 각 행이 Train / Validation / Test 중 어디인지 **1단계에서 만든 집합**으로 판별합니다.
- 각 URL에 대해:
  - **숫자 30개**: `extract_features`로 계산 (실패하면 0으로 채움).
  - **숫자 2개 (`te_tld`, `te_sld`)**:
    - **Train 행**: 2단계에서 만든 **OOF** 값.
    - **Val/Test 행**: 2단계에서 만든 **Train 전체 맵**으로 조회 (없으면 Train 전체 평균).
  - **문자 4개**: 경로·TLD·SLD·서브도메인 (`parse_url_components`).
  - **정답 라벨**도 각 집합용 배열에 넣습니다.

**왜 이렇게 하나요?**

- 1단계는 “**인덱스만**” 정했고, 3단계에서 비로소 **모델이 볼 표**를 만듭니다.
- Train에서 OOF 순서와, 여기서 Train 행을 채우는 순서가 **같아야** 하므로, 코드는 **파일 순서대로** Train만 골라 쌓는 방식으로 맞춥니다.

---

### 4단계 — 모델이 읽기 좋은 형태로 바꾸기

**하는 일**

- **문자 4개** → `BinaryEncoder`로 **0/1 조합의 숫자 열**로 바꿉니다.  
  **학습(Train) 데이터에만** 맞추고(`fit`), Val/Test는 **같은 규칙으로만** 변환(`transform`)합니다.
- **숫자 32개** (30 + TE 2) → `StandardScaler`로 **평균 0, 분산 1에 가깝게** 맞춥니다.  
  이것도 **Train에만** `fit`합니다.
- 마지막에 **숫자 블록 + 인코딩 블록**을 가로로 이어 붙여 최종 행렬을 만듭니다.

**왜 이렇게 하나요?**

- 트리 모델이라도 **문자열은 그대로** 넣기 어렵고, **고정된 숫자 행렬**이 필요합니다.
- Val/Test 통계로 맞추면 **시험 문제가 새는 것**과 같아서, 반드시 **Train 기준**으로만 맞춥니다.
- 단순히 숫자가 큰 쪽이 중요하다는 착각을 방지하기 위해 모든 숫자를 공통된 기준으로 맞춥니다.

---

### 5단계 — XGBoost 학습하고, 점수 보고, 저장하기

**하는 일**

- `XGBClassifier` **한 개**를 만들어 Train 행렬과 라벨로 학습합니다.
- 학습 중에는 **Validation** 집합으로 로그를 봅니다 (`eval_set`).
- 끝나면 **Validation**·**Test** 각각에 대해 분류 리포트, ROC-AUC, (Test는) 혼동 행렬·미탐/오탐 비율을 출력합니다.
- 모델은 `xgb_final_v1.json`, 인코더·스케일러·맵 등은 `xgb_train_artifacts.joblib`에 저장합니다.

**왜 이렇게 하나요?**

- **Validation**: 학습이 잘 되고 있는지 **중간 점검**용.
- **Test**: 튜닝에 쓰지 않은 **마지막 시험**이라, “새 URL에 얼마나 통할지”에 가깝게 보려는 용도입니다.

---

## 참고 — 미리 알아두면 좋은 것

- **URL 전처리**: `[.]`을 `.`으로 바꾸는 디팡 복구, `http` 없으면 붙이기 등은 `normalize_url`에서 합니다.
- **메모리**: 최종적으로 Train/Val/Test **행렬이 RAM에 올라갑니다**. 전체 약 700만 행이면 **큰 메모리**가 필요할 수 있습니다. 부족하면 `USE_SAMPLE`로 줄여서 먼저 돌려 보세요.
